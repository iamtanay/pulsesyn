## What Was Built

### Commit 1 — Scaffold

Full directory tree per coding standards v1.0. A `doc.go` in every package declaring its scope and dependency rules. `go.mod` initialised at `github.com/iamtanay/pulsesyn`. `README.md` and `DECISIONS.md` created.

The core isolation principle was enforced from the first commit — `core/*` packages declare no external imports in their `doc.go` files.

### Commit 2 — `core/claim`

The central protocol primitive. A `Claim` is a structured, falsifiable assertion submitted with a pointer to supporting evidence.

- `claim.go` — `Claim` struct (immutable), `ClaimType`, `GeographicScope`, `Input` constructor, `NewClaim()`, field validation against all spec §2.1.2 rules, SHA-256 claim ID derivation, domain tag normalisation
- `lifecycle.go` — `LifecycleState` type, seven states, `validTransitions` map, `Transition()` returning a new immutable `Claim`, `IsTerminal()`
- `claim_test.go` — 17 tests covering valid construction, deterministic ID, domain tag normalisation, all 11 validation error paths, all 7 valid lifecycle transitions, 5 invalid transitions, and terminal state

**Bug caught by tests:** `domain_tags_only_whitespace` was passing when it should have failed. `newClaimSkipURL` was checking `len(in.DomainTags) == 0` before normalisation — a slice of `["  ", ""]` has length 2 and passed through. Fixed by checking `len(normaliseDomainTags(in.DomainTags)) == 0` instead.

### Commit 3 — `core/consensus`

The PoV vote weighting engine. Takes a set of validator votes and produces a verdict with a confidence score.

- `result.go` — `VerdictState` (four constants), `VerdictBreakdown` (per-verdict weight totals for audit transparency), `ConsensusResult`
- `consensus.go` — `Vote` struct, `ComputeConsensus()` entry point, quorum check (2/3 participation), `aggregateWeights()`, `adjustedWeight()` formula (`confidence × domain_reputation × (1 − bias_coefficient)`), `findWinner()`, `computeConfidence()`, `validateVote()`
- `consensus_test.go` — 10 tests covering no votes error, all invalid vote field cases, quorum not met, three honest network verdict scenarios, bias correction overriding numerical majority, even split producing indeterminate, weight formula at boundary values, participation rate accuracy, all-zero-weight indeterminate

**Design question raised:** whether individual weights needed normalizing before aggregation. Late normalization is correct — the majority check is a relative comparison (`winnerMass / TotalMass`), single-pass, no pre-computation required. Documented in `DECISIONS.md`.

### Commit 4 — `core/reputation`

The reputation engine. Makes PulseSyn self-correcting over time — accurate validators gain influence, inaccurate validators lose it.

- `reputation.go` — `ValidatorRecord` struct (immutable), `ValidatorStatus`, `DomainReputation` map, `NewValidatorRecord()`, `NewGenesisValidatorRecord()` with 0.75 starting scores in registered domains, `DomainScore()`, `IsEligibleForDomain()`, `withUpdatedDomainScore()` internal helper, `computeGlobalReputation()`, `clamp()`
- `updates.go` — `VoteOutcome` input, `UpdateResult` output, `UpdateReason` constants, `ApplyPostFinalizationUpdate()` implementing all six spec §5.2 cases plus absent and late penalties, `computeDelta()`
- `decay.go` — `ApplyDecay()` implementing `new_score = old_score × (1 − 0.001)^days_inactive`. Activates after 30 days inactivity. Floor enforced at `EligibilityThreshold`.
- `reputation_test.go` — 14 tests covering constructor validation, genesis starting scores, eligibility across all threshold and status combinations, all eight update cases, ceiling and floor clamping, immutability, decay below/at/above threshold, floor invariant, global reputation average

**Bug caught by tests:** `IsEligibleForDomain` was returning `true` for scores of 0.10 and 0.0. The cause was `withUpdatedDomainScore` clamping to `ReputationFloor` (0.15) on write, silently elevating scores set below the threshold. Pushed back on whether the fix was a protocol decision or a test-passing patch — confirmed correct protocol design. `ReputationFloor` protects established validators from decay, it is not a starting baseline. New validators start at zero. Fixed by clamping to `[0.0, ReputationCeiling]` in the helper and leaving floor enforcement in `ApplyPostFinalizationUpdate` and `ApplyDecay` only. Documented in `DECISIONS.md`.

## Design Decisions

All decisions recorded in `DECISIONS.md`. Key decisions this session:

- SHA-256 used in Phase 1 instead of SHA3-256 — `golang.org/x/crypto` not yet approved as an external dependency. Documented at the call site and in `DECISIONS.md`. Must be resolved before Phase 2 chain integration.
- `ReputationFloor` enforced in update and decay operations only, not in the internal `withUpdatedDomainScore` helper.
- Late normalization in consensus — individual weights aggregated in raw weight space, normalized only at the decision boundary.
- Core isolation enforced from commit one — `core/*` imports stdlib only.
- Immutability across all protocol structs — `Claim`, `ValidatorRecord`, `ConsensusResult`. Update operations return new structs.

## What Session 2 Must Build

- `core/bias` — bias detection module. Sliding window of last N validations per validator per domain. Bias coefficient computation against population average. Response tier classification (`Negligible`, `Moderate`, `High`, `Severe`). Bias correction factor output consumed by `core/consensus`.
- `simulation/` — simulation harness. Configurable validator pool, domain distribution, collusion rate, bias distribution. Scenarios: honest network, 15% collusion, 30% collusion, high bias concentration. Validates Phase 1 protocol math at scale.
